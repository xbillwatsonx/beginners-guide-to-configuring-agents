# 04 - Maintenance

## Why maintenance matters

Your agent is not a one-time install.

Once it starts helping with real work, things change:

- memory grows
- projects move
- tools update
- providers change
- rules get added
- old assumptions become wrong

If nobody checks on the setup, the agent slowly gets less reliable.

Maintenance is how you keep the agent healthy.

## What maintenance is

Maintenance is not complicated. It means checking the basics on a schedule.

Good maintenance asks:

- Is the agent still starting correctly?
- Is the model/provider still configured the way I expect?
- Is memory still small and useful?
- Are there stale rules or old project notes?
- Are paid providers still blocked unless I approve them?
- Are important files backed up?
- Are there repeated mistakes that should become rules?

That is it. You are not trying to make the setup perfect. You are trying to catch drift before it becomes frustrating.

## The real lesson

We learned this because little issues kept turning into bigger confusion:

- one agent had memory full of reference material
- another had provider routing that needed to be locked down
- an update attempt happened before enough research was done
- old environment assumptions kept showing up in new sessions

None of these required a genius fix. They required regular checks.

## What you will build in this section

By the end of this section, you should have:

- a simple weekly health check
- a memory review habit
- a provider/cost check
- a backup habit for context files
- a prompt you can copy-paste when you want the agent to inspect itself
